WHAT JUST HAPPENED?

Optional reading. Or just ask the person at the booth.

SQL INJECTION

The database believed you

Websites query their database using a language called SQL. When your input goes directly into that query without any check, you can write part of the query yourself. Adding OR 1=1 made the condition always true, so every account came back instead of just one.

Fix: use parameterized queries so the database always treats user input as plain data, never as code.

IDOR

The server trusted the URL

The account page loaded whatever ID appeared in the URL, with no check that you actually own that account. Changing one number in your browser was enough to open anyone's account page. This is called Insecure Direct Object Reference.

Fix: always verify on the server that the logged-in user owns the resource they are requesting.

BROKEN ACCESS CONTROL

The server trusted the form

HTML forms can contain hidden fields that are invisible on screen but still sent to the server. The PIN check used an account ID from one of those fields. Since you could edit it in the browser inspector, you swapped it for your own account ID and used your own PIN. The server never noticed.

Fix: never make security decisions based on data the client can modify. Use the server session instead.
