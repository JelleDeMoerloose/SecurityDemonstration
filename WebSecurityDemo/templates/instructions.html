01
SQL INJECTION

FIND CHARLIE'S ACCOUNT ID

The bank has a Find User feature, used to look up account IDs before transferring money. Try typing a number like 1 or 2 and see what comes back.

Behind the scenes, your input gets placed directly into a database command that looks like this:

# The query the server runs when you type "1" SELECT username, account_id FROM users WHERE accounts.user_id = 1

The database reads this as: "give me the user where the ID equals 1" and returns exactly that one person. But in database language, you can chain extra conditions using OR, just like in plain English. And the developer forgot to limit what you can type into that field.

Your challenge: add an OR condition to the search input that is always true, so the database ignores the filter and returns every user at once.

💡 Hint: Is 1=1 always true? Try adding that after an OR in the input field. Once you see all users listed, find Charlie and note his account ID for the next step.

02
IDOR — INSECURE DIRECT OBJECT REFERENCE

BREAK INTO CHARLIE'S ACCOUNT

Now that you have Charlie's account ID, navigate to your My Account page. This is the page where you can transfer money to other accounts.

Have a good look around the page. Does anything give away how the bank keeps track of which account it is showing you? Think about what information the browser needs to display the right account, and where that information might live.

Once you figure that out, ask yourself: what would happen if that value belonged to someone else?

💡 Hint: Look at the URL and make changes to it. A secure server should always check whether the logged in user actually owns the account they are requesting. This one does not. It simply loads whatever account ID is given to it, no questions asked.

03
BROKEN ACCESS CONTROL — HIDDEN FIELD MANIPULATION

BYPASS THE PIN VERIFICATION

You are on Charlie's account page. Fill in the transfer form and try to send €500,000 to yourself. But the form asks for a PIN and you have no idea what Charlie's PIN is. Dead end?

Here is something most people do not know: every form on a website is built out of small pieces of code called fields. Most fields are visible, like the amount you type in or the PIN box. But forms can also contain hidden fields that are invisible on the page yet still get sent to the server when you click submit.

You can see all of this hidden code yourself. Right-click anywhere on the transfer form and click Inspect. A panel opens showing the code behind the page. Look through the transfer form code and see if you can spot a hidden field. It will look something like this:

<input type="hidden" name="account_id" value="...">

Found it? The server uses that value to decide which account's PIN to check when you hit submit. You know the PIN of one account. Can you make the server check that one instead of Charlie's?

💡 Hint: Double-click the value in the inspector to edit it, change it to your own account ID (10), then submit the form using your own PIN= 1234. 💸

◈ WHAT YOU JUST EXPLOITED

SQL INJECTION (OWASP A03)

Unsanitized user input injected directly into a database query. Fix: use parameterized queries / prepared statements.

IDOR (OWASP A01)

Server exposes internal object references without verifying ownership. Fix: always check if the logged-in user owns the requested resource.

BROKEN ACCESS CONTROL (OWASP A01)

Server trusts client-supplied hidden fields to determine which account to verify. Fix: derive account identity from the server session, never from form data.
